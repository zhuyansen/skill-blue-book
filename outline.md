## Part 1｜基础：Skill 是什么

### 第 1 章 · 为什么需要 Skill：从 Mahesh 到 Barry

**核心**：Anthropic 的"Mahesh vs Barry"类比——聪明的大模型 vs 有经验的领域专家。这一章讲 Skill 要解决的本质问题：**Agent 的"每次从零开始"**。

**独特角度**：除了 Anthropic 的原论述，加一层："为什么在 2026 年 Q1 才爆发"——答案是 Claude Code + Cursor + Codex 同时把 skill loading 做成一等公民，形成格式化压力。

**字数估算**：4,000-5,000 字。

---

### 第 2 章 · 三层渐进加载：Skill 的真正魔法

**核心**：description → instructions → resources 的三层按需加载机制。这是 Skill 能无限扩展而不爆上下文的技术底座。

**独特角度**：把 Skill 对比 System Prompt / RAG / Tool Use / MCP Server 五种方案的上下文消耗量。用 AgentSkillsHub 的真实 skill 做例子，画一张"每层消耗多少 token"的图。

**字数估算**：4,500-5,500 字 + 1-2 张架构图。

## Part 2｜实战：怎么写好 Skill

### 第 4 章 · 站在 Agent 角度设计 Skill（宝玉四条哲学）

**核心**：复盘宝玉的 4 条设计哲学——Agent 视角、原子化、Skill 自我迭代、脚本优先。每条配 2-3 个真实 skill 案例。

**独特角度**：用 Hub 数据验证"原子化"是否真的产出更高质量——筛选 Top 100 skill，看它们平均有多"原子"（通过 README 长度、脚本数、function 数量）。

**字数估算**：6,000-7,000 字 + 1 张"原子化 vs 万能化"对比表。

---

### 第 5 章 · 迭代优化的闭环：从踩坑到飞轮

**核心**：Anthropic 官方"评估驱动开发"四步 loop + 宝玉的"Agent 自我优化"机制。这是让 Skill 从"写完就扔"进化到"越用越好"的关键。

**独特角度**：用 Hub 的 star velocity 数据画"持续迭代 vs 一次性发布"的生存曲线。预计结论：持续每周 commit 的 skill，2 年后存活率 >70%；一次性发布的 <20%。

**字数估算**：5,500-6,500 字 + 1 张生存曲线图。

## Part 3｜生态：Skill 正在吞噬一切

### 第 7 章 · 四大框架的对标与选择

**核心**：anthropic/skills（标准化工具箱）vs obra/superpowers（强制方法论）vs garrytan/gstack（虚拟团队）vs compound-engineering（知识飞轮）。四条路线的技术选型和使用场景。

**独特角度**：实测 4 个框架做同一个任务（"帮我审阅一段代码"），比较：启动速度、上下文占用、结果质量、可定制性。这是业内第一次有人做横向跑分。

**字数估算**：6,000-7,000 字 + 1 张对比 matrix + 4 段跑分截图。

---

### 第 8 章 · Skill 正在吞噬其他柱子

**核心**：Skill 本来只是"能力扩展"，但 superpowers 把 Harness（工作流约束）吞了，gstack 把 Memory（持久化经验）吞了，compound-engineering 把 Safety（代码审查）吞了。Skill 正变成 Agent 系统的中枢。

**独特角度**：画一张"Skill 的疆域扩张"的 Mermaid 图——从 2024 年的"prompt 片段"到 2026 年的"中枢"。预测 2027 的形态（带 retrieve、evaluate、self-improve 的闭环）。

**字数估算**：5,500-6,500 字 + 演化图。

---

### 第 9 章 · Distribution：商业化三角少的那条边

**核心**：直接承接 lovstudio《商业化三角不可能定理》。三角是：标准化 Runtime × 本地运行 × 源码保护。我的论点：**不可能三角是对的，但大多数作者卡在进入三角之前——找用户**。

**独特角度**：
- 用 Hub 数据证明"找不到用户"是真实问题（参考第 3 章 54% 0 star）
- 三条路径（Hosted / Closed Runtime / Service-on-Open）+ **Distribution 这条被忽视的边**
- 解释为什么 Hub 要承担 Distribution 这块——不是竞争 Anthropic，是补位

**字数估算**：6,500-7,500 字 + 三角+第四条边的图。

## Part 4｜实践：AgentSkillsHub 运营手记

### 第 10 章 · Verified Creator：不是花钱买的认证 ✅

**状态**：已完稿（~6,500 字 + 评估公式 + 完整申请表模板）

### 第 11 章 · 咨询撮合 + 企业目录：Service-on-Open 怎么跑

**核心**：第 9 章的"Service-on-Open"路径具体怎么跑。两个产品：
- **Skill 咨询撮合**：作者挂出咨询报价，Hub 撮合，抽 10% 成交费
- **企业订阅目录**（Starter ¥9,999 / Business ¥29,999 / Enterprise 定制）：合规筛选过的 Skill 目录 + 安全审计报告

**独特角度**：
- 给 skill 作者的**收入路径模拟**（假设月访问 1K，转化 2% 咨询，每单 2000 元 → 年化 48 万）
- 第一个企业客户的完整接洽过程（如果到写作时已经签约）
- 价格表的推导逻辑（不拍脑袋）

**字数估算**：5,500-6,500 字 + 定价推导 + 作者分成协议样本。

**核心**：Anthropic 原文里有一句"当 Claude 自己开始创建 Skills 的时候，系统会真正转起来"。这章讨论：**如果 Claude 自产 Skill 成为现实，人类作者的位置在哪？**

**独特角度**：

- 给每一类作者（从一次即弃到 Top 1%）的 2026-2028 生存指南

## 风险和取舍

1. **数据的时效性**：蓝皮书写 6 周，Hub 数据持续变化。解决：每章开头明确数据快照时间；GitHub repo 长期维护；每季度更新。
2. **引用 lovstudio、宝玉、花叔的论点不能断章取义**：解决：每次引用必须原文 + 指向原文链接；写完发 DM 给本人核稿。
3. **"自我解剖"会不会显得软弱**：不会。lovstudio 的原文就是自我解剖式的爆款。诚实是内容最稀缺的资源。
4. **写完没人看**：解决：每 2 章一条 X 长推文 + 公众号双开。X 是种子，公众号是扩音。
